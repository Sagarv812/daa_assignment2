\section*{Abstract}

This report explains the current cleaned implementation of the assignment solver in the submitted codebase. The program reads a polygon with holes, converts it into a single merged boundary using bridge insertion, decomposes the merged boundary into monotone pieces, triangulates those pieces, and finally computes a constructive guard set by 3-coloring the triangulation graph. The report focuses only on the submitted files in \submission. It does not rely on auxiliary scripts or named testcase files, and all worked examples are described directly in geometric terms.

\section{Assignment Goal}

The input is a polygonal gallery consisting of one outer boundary and zero or more interior holes. Let:

\begin{itemize}
\item $n$ be the total number of input vertices across the outer boundary and all holes,
\item $h$ be the number of holes.
\end{itemize}

The solver must:

\begin{enumerate}
\item triangulate the walkable region,
\item compute a valid point-guard set,
\item print the triangles and guard coordinates.
\end{enumerate}

The implementation is organized around two formulas:

\[
\text{number of triangles} = n + 2h - 2
\]

\[
\text{guard upper bound} = \left\lfloor \frac{n+h}{3} \right\rfloor
\]

The first formula is a correctness target for triangulation. The second formula gives the theoretical upper bound used by the guard stage.

\section{End-to-End Pipeline}

The solver is easiest to understand as a sequence of seven stages. The first three stages keep the gallery in half-edge form. After holes are bridged, the triangulation code works on a merged polygon walk indexed by occurrences along that walk.

\begin{figure}[H]
\centering

The design choice that drives the whole solution is this:

\begin{quote}
do the topological work first by removing holes through bridges, then run standard polygon triangulation ideas on the resulting merged boundary.
\end{quote}

That keeps the difficult part localized inside the hole-merging phase and lets the later stages use well-known sweep-line and graph-coloring techniques.

\subsection{Orientation Invariant}

The parser immediately normalizes orientation:

\begin{itemize}
\item the outer boundary is stored counter-clockwise,
\item every hole is stored clockwise.
\end{itemize}

This invariant matters for three reasons:

\begin{itemize}
\item local cross-product tests become consistent,
\item helper and bridge directions are interpreted correctly,
\item the interior stays on the expected side of the walk.
\end{itemize}

\subsection{Parsing Logic}

For each cycle, \code{src/parser.cpp} does the following:

\begin{enumerate}
\item read all coordinates into a temporary vector,
\item compute the signed area,
\item reverse the order if the orientation is opposite to the required convention,
\item allocate one \code{Vertex} and one \code{HalfEdge} per coordinate,
\item connect the half-edges into a circular doubly linked list.
\end{enumerate}

After parsing is complete, \code{src/polygon_utils.cpp} supplies the shared traversal helpers used later by hole merging and guard computation:

\begin{itemize}
\item \code{addVertices} walks a cycle and collects its vertices,
\item \code{getTopmostVertices} extracts the topmost vertex of each hole,
\item \code{getTotalVertexCount} counts the original input vertices before topology changes happen.
\end{itemize}

\section{Hole Merging}

\subsection{Why Hole Merging Comes First}

The triangulation algorithm used in the submission expects a single polygonal walk. A gallery with holes is therefore converted into one merged boundary before triangulation begins.

The implementation in \code{src/hole_merger.cpp} performs a top-down sweep over all vertices. The key event is the moment when the sweep reaches the topmost vertex of a hole. At that moment, the hole is connected to the nearest valid wall on its left.

\subsection{Three Tracking Structures Drive the Merge}

The updated hole merger is no longer just ``active edges plus one helper vertex.'' It maintains three synchronized structures because the sweep now has to distinguish geometry, local topology, and boundary-component ownership:

\begin{enumerate}
\item \textbf{Active-edge map.} A balanced map from active wall edges to full \code{HelperState} records. The key order comes from sweep-line intersection on the current horizontal ray.
\item \textbf{Outgoing-angle index.} A per-vertex map of outgoing half-edges ordered by direction angle is used to choose the exact splice location after a target vertex has been selected.
\item \textbf{Component map.} A table from each original \code{Vertex*} to its source component records whether that vertex belongs to the outer boundary or to one specific hole cycle.
\end{enumerate}

The three structures answer three different questions:

\begin{itemize}
\item which wall is immediately to the left of the current sweep event,
\item which helper information on that wall should survive same-level updates,
\item where exactly the bridge should be spliced around the chosen target vertex.
\end{itemize}

The new \code{HelperState} structure is the main change in the current \code{src/hole_merger.cpp}.

The helper-related support functions now divide the update logic cleanly:

\begin{itemize}
\item \code{makeUnlockedHelper} and \code{setUnlockedHelper} create ordinary unlocked helper states,
\item \code{setHoleTopHelper} locks an active wall to a hole-top event,
\item \code{canOverwriteWithOrdinary} decides whether an ordinary sweep vertex may replace a locked helper,
\item \code{applyOrdinaryHelperUpdate} updates the helper state while preserving same-level extrema,
\item \code{getBridgeTarget} chooses which locked helper endpoint should actually receive the bridge.
\end{itemize}

Two small implementation details matter here. First, the component table is built once before the sweep so that every overwrite decision can check component ownership in constant time. Second, the outgoing-angle index uses an \code{OutgoingKey} made from the outgoing direction angle and a unique edge id, so multiple outgoing edges with the same angle can still coexist in the map without collisions.

\subsection{Why the Same-Level Lock Exists}

The lock solves a subtle sweep-order problem. Suppose a hole-top event occurs at height $y_0$, and several vertices on the left wall also lie on that same height. The code wants all same-level vertices from the \emph{same} boundary component to be treated as one logical helper region, but it does not want another component at the same height to overwrite that state before the bridge choice is finalized.

That is exactly what the lock enforces:

\begin{itemize}
\item if a candidate vertex is strictly below the locked height, the helper can be replaced normally,
\item if a candidate vertex is at the same height, it may refine the helper only if it comes from the same component,
\item while that happens, the code remembers both the minimum-$x$ and maximum-$x$ helper on the locked level.
\end{itemize}

The implementation compares heights using a small tolerance \code{kHelperLockEps = 1e-9}. In effect, the code treats vertices on the same geometric level as belonging to one locked band, then filters that band by component id.

When a later hole top queries the same wall, \code{getBridgeTarget()} compares the query vertex against those two stored extremes and chooses the closer one in $x$.

\subsection{Actual Merge Sequence in the Code}

The current code follows this sequence:

\begin{enumerate}
\item Use \code{getTopmostVertices} to select one bridge source per hole.
\item Sort all vertices from top to bottom, breaking equal-$y$ ties by smaller $x$.
\item Initialize the outgoing-angle index with the outer cycle and every hole cycle.
\item Build \code{vertexComponentIds} so every original vertex knows which boundary component it belongs to.
\item Sweep through the sorted vertices.
\item Before modifying the sweep state at a vertex, remember the current closest left wall as \code{preLeftWall}.
\item If the current vertex is the next hole-top vertex, use the stored helper state of \code{preLeftWall} to choose the actual target vertex on that wall.
\item Use the outgoing-angle index to choose a valid outgoing half-edge at the target vertex.
\item Insert \code{bridge} and \code{bridgeTwin}, then rewire only the nearby \code{nextEdge}/\code{prevEdge} links.
\item Update active edges for the current vertex; newly active edges receive ordinary unlocked helper states.
\item Recompute the closest left wall after those updates.
\item If a hole top was processed, lock the helper state on the relevant active wall using \code{setHoleTopHelper}.
\item If no hole top was processed, first test whether an ordinary overwrite is allowed.
\item Only then apply the ordinary helper update.
\item If the pre-bridge left wall is still active after the bridge update, relock it so the hole-top information is not lost when the active-edge neighborhood changes.
\end{enumerate}

This logic appears in the following key functions:

\begin{itemize}
\item \code{getRayIntersectionX}: computes the $x$-coordinate where an edge meets the sweep line,
\item \code{findClosestWallToLeft}: binary-searches the active-edge map,
\item \code{addCycleVerticesToComponentMap}: records the source component of every original vertex,
\item \code{getBridgeTarget}: chooses a bridge endpoint from the locked helper state,
\item \code{findValidSplicePoint}: chooses the correct outgoing edge at the target vertex,
\item \code{buildBridge}: allocates the bridge pair and rewires local topology.
\end{itemize}

\subsection{Why the Target Vertex Needs Angular Disambiguation}

Once one bridge has already been inserted, a later bridge may hit a vertex that now has multiple outgoing edges. If the new bridge is inserted against the wrong outgoing edge, the merged walk becomes topologically incorrect even if the bridge endpoints themselves were geometrically sensible.

That is why the code uses:

\begin{itemize}
\item bridge direction angle,
\item outgoing half-edge angle,
\item a sector-membership test.
\end{itemize}

Together, these checks make the splice robust against earlier modifications around the target vertex. The ordered outgoing map stores keys of the form \code{(angle, edgeId)}, and \code{findValidSplicePoint()} probes that map with \code{upper\_bound}. This lets the code locate the outgoing edge immediately preceding the desired bridge direction and then verify that the bridge lies inside the correct local sector.

\subsection{State After Merging}

After all holes are processed:

\begin{itemize}
\item the outer boundary and holes are no longer separate cycles,
\item \code{gallery->boundaryEdge} reaches one merged boundary walk,
\item some original vertices can now appear more than once when that merged walk is traversed.
\end{itemize}

That last detail is crucial. The triangulation stage does not treat the merged boundary as a set of unique coordinates. It treats it as a sequence of \emph{occurrences} along a walk.

\section{Triangulation}

\subsection{Merged Walk as an Occurrence-Indexed Polygon}

The current version of \code{src/triangulation.cpp} first traverses the merged boundary and copies each visited vertex into a \code{PolygonVertex} record:

\begin{itemize}
\item the coordinates are copied directly,
\item the original \code{Vertex*} is preserved,
\item the position in the traversal becomes the occurrence index.
\end{itemize}

This is a deliberate design choice. The code no longer perturbs coordinates to separate repeated bridge endpoints. Instead, it distinguishes occurrences combinatorially by their position in the merged walk.

\begin{figure}[H]
\centering
\includegraphics[width=0.94\textwidth]{occurrence_walk}
\caption{Why occurrence indices matter after bridging. The same original geometric vertex can appear more than once in the merged walk.}
\label{fig:occurrence-walk}
\end{figure}

This is also why the triangulation result stores both:

\begin{itemize}
\item \code{triangleIndices}: triangles in occurrence space,
\item \code{occurrenceVertices}: a map back from occurrence indices to original \code{Vertex*} pointers.
\end{itemize}

\subsection{Vertex Classification}

Every occurrence is classified as one of the six standard sweep-line categories:

\begin{itemize}
\item \code{START},
\item \code{END},
\item \code{SPLIT},
\item \code{MERGE},
\item \code{REGULAR_LEFT},
\item \code{REGULAR_RIGHT}.
\end{itemize}

The classification uses:

\begin{itemize}
\item relative sweep order of the previous and next vertices,
\item the sign of the orientation test around the current vertex.
\end{itemize}

In exact code terms, the six outcomes are:

\begin{itemize}
\item \code{START}: \code{prevBelow} and \code{nextBelow} are both true, and \code{ccw(prev, curr, next) > eps}.
\item \code{SPLIT}: \code{prevBelow} and \code{nextBelow} are both true, and \code{ccw(prev, curr, next) <= eps}.
\item \code{END}: \code{prevBelow} and \code{nextBelow} are both false, and \code{ccw(prev, curr, next) > eps}.
\item \code{MERGE}: \code{prevBelow} and \code{nextBelow} are both false, and \code{ccw(prev, curr, next) <= eps}.
\item \code{REGULAR_LEFT}: the vertex is in the mixed case and \code{isAbove(prev, curr)} is true.
\item \code{REGULAR_RIGHT}: the vertex is in the mixed case and \code{isAbove(prev, curr)} is false.
\end{itemize}

\subsection{Monotone Decomposition}

The function \code{makeMonotone} builds a temporary sweep boundary and then processes occurrence vertices from top to bottom. Its active-edge structure stores helper occurrence indices rather than raw geometry objects. The diagonal rules follow the standard monotone-decomposition sweep:

\begin{itemize}
\item start vertices insert their outgoing edge,
\item end vertices may connect to a merge helper on the incoming edge,
\item split vertices connect to the helper of the closest left wall and then insert the outgoing edge,
\item merge vertices may need one diagonal to the incoming-edge helper and another to the closest left-wall helper,
\item regular vertices either update the left-wall helper or replace the incoming edge by the outgoing edge, depending on which chain they lie on.
\end{itemize}

The current implementation also has several practical filters in \code{addDiagonal}. These are necessary because repeated bridge endpoints can create multiple occurrences with identical coordinates.

\begin{itemize}
\item A vertex is never connected to itself.
\item Adjacent polygon vertices are never treated as diagonals.
\item Two occurrences with the same coordinates are not connected directly.
\item Existing polygon edges are not re-added as diagonals.
\item Duplicate undirected diagonals are ignored.
\end{itemize}

\subsection{Recovering Faces Without Building Another DCEL}

After monotone diagonals are known, the program still needs the actual monotone faces. The code solves that problem in a distinctive way:

\begin{enumerate}
\item place the occurrence indices on a synthetic circle,
\item add the polygon boundary edges and the diagonals into one undirected graph,
\item sort each adjacency list by angular order on that circle,
\item walk directed edges while always turning to the previous angular neighbor,
\item keep only the positive-area cycles.
\end{enumerate}

Two details matter for correctness here:

\begin{itemize}
\item the embedding is the same one built by the code, namely occurrence $i$ placed at angle $2\pi i / m$,
\item the diagonal set must be outerplanar, so the illustrative chords must be noncrossing.
\end{itemize}

\begin{figure}[H]
\centering
\includegraphics[width=0.72\textwidth]{outerplanar_embedding}
\caption{Synthetic circular embedding used in \code{extractMonotoneFaces}. Left: a noncrossing outerplanar graph on the same counter-clockwise circular placement used by the code. Right: a local example of the actual traversal rule, where arriving along \code{0 -> 4} makes the walk continue along \code{4 -> 5} because vertex 5 is the previous angular neighbor of 0 in the sorted rotation order around 4.}
\label{fig:outerplanar}
\end{figure}

This is one of the strongest parts of the implementation because it avoids introducing another heavy topology structure while still recovering faces reliably from the diagonal graph.

\subsection{Triangulating Each Monotone Face}

For every extracted monotone face, \code{triangulateMonotoneFace} runs the standard stack-based algorithm:

\begin{enumerate}
\item identify the top and bottom vertices,
\item mark the left chain and right chain,
\item process vertices in sweep order,
\item pop from the stack while the local chain test says a diagonal is valid,
\item add the final diagonals to close the fan at the last vertex.
\end{enumerate}

Then the code extracts faces one more time. Any face with exactly three non-degenerate vertices is accepted by \code{appendTriangleIfValid}. This function also normalizes orientation so the printed triangle order is consistent.

\subsection{Important Cleaned-Code Details}

The current cleaned triangulation code contains a few details that are easy to miss but important to explain:

\begin{itemize}
\item \code{buildMergedPolygon} now uses the merged walk directly rather than inventing perturbed coordinates.
\item \code{simplifyFace} removes duplicate consecutive occurrences but intentionally preserves genuine collinear occurrences, because different bridge events may need those vertices to remain explicit.
\item \code{appendTriangleIfValid} rejects zero-area triangles before they reach output.
\item \code{printTriangles} prints exactly the triangle vertex coordinates expected by the assignment output format.
\end{itemize}

\section{Guard Computation}

\subsection{Bound and Graph Construction}

The guard stage uses the original input counts recorded \emph{before} hole merging:

\[
\texttt{theoreticalBound} = \left\lfloor \frac{n+h}{3} \right\rfloor
\]

This value is stored in \code{GuardSolution}, but the solver does not blindly print that bound. Instead, it tries to construct a smaller valid set when possible.

The graph built by \code{src/guards.cpp} has:

\begin{itemize}
\item one graph vertex per occurrence in the triangulation,
\item one edge between every pair of occurrence indices that share a triangle.
\end{itemize}

Because the graph comes from a triangulated polygonal subdivision, it is outerplanar.

\subsection{Reverse-Degeneracy 3-Coloring}

The function \code{colorOuterplanarGraph} colors the graph in two passes:

\begin{enumerate}
\item repeatedly remove a vertex of degree at most two, recording the removal order,
\item traverse that order backward and color each vertex with the first available color.
\end{enumerate}

This works because outerplanar graphs are 2-degenerate, so the forward elimination always succeeds on a correct outerplanar triangulation.

\begin{figure}[H]
\centering
\includegraphics[width=0.78\textwidth]{guard_coloring_clean}
\caption{A code-faithful coloring example. The red class is uniquely smallest after deduplicating original \code{Vertex*} pointers, so it is the one selected by the current implementation.}
\label{fig:guard-coloring}
\end{figure}

After coloring, the code:

\begin{itemize}
\item gathers original \code{Vertex*} pointers by color class,
\item deduplicates them with ordered sets,
\item selects the smallest color class as the guard set.
\end{itemize}

This is why the actual guard count printed by the program can be smaller than the theoretical bound.

If two color classes tie after deduplication, the current code keeps the earliest minimum it encounters because \code{bestColor} is updated only on a strict improvement.

\subsection{Fallback Behavior}

If the coloring step somehow fails, the code falls back to the first distinct original vertices it can collect, up to the theoretical bound. This fallback is conservative, but it keeps the solver from producing an empty answer in the presence of an unexpected coloring failure.

\section{Complexity Analysis and Engineering Notes}

\subsection{Notation Used for the Analysis}

Let:

\begin{itemize}
\item $n$ be the total number of original input vertices,
\item $h$ be the number of holes,
\item $m$ be the number of vertex \emph{occurrences} on the merged walk after hole bridging.
\end{itemize}

In this implementation, each hole bridge causes two bridge endpoints to reappear in the merged boundary traversal, so:

\[
m = n + 2h
\]

This is consistent with the triangle formula for a simple polygon obtained after bridging:

\[
m - 2 = n + 2h - 2
\]

So the later triangulation and guard stages are most naturally analyzed in terms of $m$, and then converted back to $(n,h)$ at the end.

\subsection{Parsing and Utility Routines}

\textbf{Parsing one test case.}
\begin{itemize}
\item Reading all coordinates takes $\bigO{n}$ time.
\item Computing signed area for each cycle is linear in that cycle size, so the total is still $\bigO{n}$.
\item Allocating one \code{Vertex} and one \code{HalfEdge} per input vertex is $\bigO{n}$.
\item Linking each cycle into a circular doubly linked list is $\bigO{n}$.
\end{itemize}

So the full parsing stage costs $\bigO{n}$ time and $\bigO{n}$ space.

\textbf{Utility routines in \code{src/polygon_utils.cpp}.}
\begin{itemize}
\item \code{addVertices}: $\bigO{k}$ for a cycle of length $k$.
\item \code{getMergedPolygonVertices}: $\bigO{m}$ after hole merging.
\item \code{getTopmostVertices}: scans every hole cycle once, so $\bigO{n}$ total before merging.
\item \code{getTotalVertexCount}: scans every cycle once, so $\bigO{n}$.
\item \code{getVerticesSorted}: collect all vertices in $\bigO{n}$ and sort them in $\bigO{n \log n}$, giving $\bigO{n \log n}$ overall.
\end{itemize}

\subsection{Hole Merging Complexity}

The updated hole-merging phase has five distinct costs.

\textbf{1. Topmost-hole extraction.}

\code{getTopmostVertices} scans all hole cycles once, so this part is $\bigO{n}$.

\textbf{2. Sweep-order preparation.}

All original vertices are collected and sorted by sweep priority, costing $\bigO{n \log n}$.

\textbf{3. Index initialization before the sweep.}

Two auxiliary structures are prepared before the first sweep event:

\begin{itemize}
\item the outgoing-angle index inserts one initial outgoing edge per original vertex, costing $\bigO{n}$ total,
\item \code{addCycleVerticesToComponentMap} records one component id per original vertex, also costing $\bigO{n}$ total.
\end{itemize}

\textbf{4. Main sweep over vertices.}

The sweep visits every original vertex once. For each event it performs only a constant number of:

\begin{itemize}
\item active-edge map lookups,
\item active-edge insertions and deletions,
\item \code{upper_bound} queries,
\item helper-state updates.
\end{itemize}

The helper-state routines such as \code{setHoleTopHelper}, \code{canOverwriteWithOrdinary}, and \code{applyOrdinaryHelperUpdate} are constant-time because they only read or write a few fields, including the tracked minimum and maximum same-level helpers. The active-edge map is still a balanced \code{std::map}, so the dominant per-event cost is $\bigO{\log n}$. Therefore the full sweep contributes $\bigO{n \log n}$.

\textbf{5. Hole-top bridge events.}

When a hole topmost vertex is encountered, the extra work is:

\begin{itemize}
\item one active-edge lookup for the pre-bridge left wall,
\item one constant-time \code{getBridgeTarget} decision from the stored helper extrema,
\item one search in the outgoing-angle index, costing $\bigO{\log d(v)}$ where $d(v)$ is the number of outgoing edges currently stored for the chosen target vertex,
\item constant-time pointer rewiring in \code{buildBridge},
\item two outgoing-edge insertions,
\item one extra active-edge lookup to preserve the pre-bridge wall lock if that wall remains active.
\end{itemize}

Across all holes this remains within $\bigO{h \log n}$ worst-case time, which is still dominated by the global sort and the sweep itself.

\textbf{Hole-merging total.}

\[
\text{Time} = \bigO{n \log n}, \qquad \text{Space} = \bigO{n + h}
\]

The extra space is due to:

\begin{itemize}
\item the sorted vertex list,
\item the active-edge map, now storing one \code{HelperState} per active wall,
\item the outgoing-angle index,
\item the component-id map,
\item the bridge half-edges added for each hole.
\end{itemize}

\subsection{Triangulation Complexity}

The triangulation stage is best analyzed in terms of the merged-walk length $m = n + 2h$.

\textbf{1. Merged polygon construction.}

\begin{itemize}
\item \code{buildMergedPolygon} traverses the merged boundary once: $\bigO{m}$ time.
\item Storing occurrence vertices also takes $\bigO{m}$ space.
\item Orientation correction by a polygon-area computation is another $\bigO{m}$.
\end{itemize}

\textbf{2. Vertex classification.}

Each occurrence is classified using only neighboring occurrences and a constant amount of arithmetic, so the total is $\bigO{m}$ time and $\bigO{m}$ space for the category array.

\textbf{3. Monotone decomposition.}

Inside \code{makeMonotone}:

\begin{itemize}
\item sorting occurrence indices by sweep order costs $\bigO{m \log m}$,
\item building the temporary sweep boundary costs $\bigO{m}$,
\item the active-edge sweep performs a constant number of balanced-map operations per occurrence, costing $\bigO{m \log m}$ overall,
\item diagonal deduplication uses a hash set, so insert/check operations are $\bigO{1}$ average per diagonal.
\end{itemize}

So \code{makeMonotone} runs in $\bigO{m \log m}$ time and uses $\bigO{m}$ extra space.

\textbf{4. Face extraction after adding diagonals.}

The function \code{extractMonotoneFaces} does:

\begin{itemize}
\item build the synthetic circular embedding: $\bigO{m}$,
\item build adjacency from polygon edges and diagonals: $\bigO{m}$ because the subdivision is planar and has only linear-size edge sets,
\item sort each adjacency list by angle: worst-case $\bigO{m \log m}$ over all vertices,
\item walk each directed edge once to recover faces: $\bigO{m}$.
\end{itemize}

Therefore one call to \code{extractMonotoneFaces} is $\bigO{m \log m}$ time and $\bigO{m}$ space in the worst case.

\textbf{5. Triangulating monotone faces.}

For a monotone face of size $s$, \code{triangulateMonotoneFace} performs:

\begin{itemize}
\item face simplification in $\bigO{s}$,
\item top/bottom detection in $\bigO{s}$,
\item sorting that face's vertices by sweep order in $\bigO{s \log s}$,
\item stack-based triangulation in $\bigO{s}$.
\end{itemize}

Across all monotone faces, the total size of all face boundaries is linear in the subdivision size, so the aggregate worst-case cost is:

\[
\bigO{m \log m}
\]

\textbf{6. Final triangular-face extraction and output assembly.}

After all triangulation diagonals are added, the code calls \code{extractMonotoneFaces} one more time and then runs \code{appendTriangleIfValid} on every face. That contributes another $\bigO{m \log m}$ worst-case time and $\bigO{m}$ space.

\textbf{Triangulation total.}

\[
\text{Time} = \bigO{m \log m}, \qquad \text{Space} = \bigO{m}
\]

Substituting $m = n + 2h$, this becomes:

\[
\text{Time} = \bigO{(n+2h)\log(n+2h)}, \qquad \text{Space} = \bigO{n+2h}
\]

\subsection{Guard Computation Complexity}

Let $t$ be the number of output triangles. Since $t = m - 2$, we have $t = \Theta(m)$.

\textbf{1. Building the triangulation adjacency graph.}

Each triangle contributes exactly three undirected edges, so \code{buildAdjacency} runs in $\bigO{t} = \bigO{m}$ time and uses $\bigO{m}$ space.

\textbf{2. Reverse-degeneracy coloring.}

The queue-based elimination phase touches each graph vertex once and each adjacency relation a constant number of times, so it is $\bigO{m}$. The reverse coloring pass is also $\bigO{m}$.

Thus the coloring procedure itself is:

\[
\text{Time} = \bigO{m}, \qquad \text{Space} = \bigO{m}
\]

\textbf{3. Deduplicating original vertices by color class.}

After coloring, the code inserts original \code{Vertex*} pointers into three \code{std::set} containers. Since ordered-set insertion is $\bigO{\log m}$ and there are $\bigO{m}$ occurrence insertions, this step is $\bigO{m \log m}$ in the worst case.

\textbf{4. Fallback path.}

If coloring fails, the fallback builds one \code{std::set<Vertex*>} from all occurrence vertices, which is also $\bigO{m \log m}$ worst-case.

\textbf{Guard-stage total.}

Because of the ordered-set deduplication used in the current code, the overall guard stage is:

\[
\text{Time} = \bigO{m \log m}, \qquad \text{Space} = \bigO{m}
\]

If the deduplication containers were changed from \code{std::set} to hashing, this stage could be reduced to linear time in practice. But for the current submitted code, $\bigO{m \log m}$ is the correct worst-case bound.

\subsection{Printing Complexity}

\begin{itemize}
\item Printing triangles costs $\bigO{t} = \bigO{m}$.
\item Printing guards costs $\bigO{g}$ where $g$ is the number of selected guards, and trivially $g \le m$.
\end{itemize}

So the output stage is linear in the size of the produced answer.

\subsection{Overall Complexity of the Full Algorithm}

Putting all stages together:

\begin{itemize}
\item parsing and utility preparation: $\bigO{n \log n}$ worst-case because of sorting,
\item hole merging: $\bigO{n \log n}$,
\item triangulation on the merged walk: $\bigO{m \log m}$,
\item guard computation: $\bigO{m \log m}$,
\item printing: $\bigO{m}$.
\end{itemize}

Since $m = n + 2h$, the full solver runs in:

\[
\text{Time} = \bigO{(n+2h)\log(n+2h)}
\]

and uses:

\[
\text{Space} = \bigO{n+2h}
\]

If the standard assignment assumption is used that $h \le n$, then this simplifies to:

\[
\text{Time} = \bigO{n \log n}, \qquad \text{Space} = \bigO{n}
\]

So the end-to-end implementation remains in the intended near-optimal sweep-line complexity class while still handling hole bridging, repeated bridge endpoints, face extraction, triangulation, and guard construction.

Several engineering decisions in the code deserve explicit mention:

\begin{itemize}
\item \code{src/main.cpp} records \code{totalVertices} and \code{holeCount} before merging so the theoretical guard bound uses the original gallery, not the modified merged walk.
\item \code{mergeHoles} updates \code{originatingEdge} at bridge endpoints so later traversal naturally follows the merged topology.
\item The updated hole merger keeps component-aware helper locks, so same-level sweep events from another boundary component cannot accidentally overwrite the helper information that a hole-top bridge decision still depends on.
\item The triangulation stage uses occurrence indices instead of trying to geometrically separate repeated bridge endpoints.
\item The guard stage prints the actual number of selected guards, not the theoretical upper bound.
\end{itemize}

\section{File-by-File Walkthrough}

\subsection{Main Driver}

\code{src/main.cpp} is short but important because it defines the exact execution order:

\begin{enumerate}
\item parse one gallery,
\item count original vertices,
\item count holes,
\item merge holes,
\item triangulate the merged gallery,
\item compute guards from the triangulation result,
\item print triangles and guards.
\end{enumerate}

The sequence is correct: both the triangle formula and the guard upper bound depend on the original gallery counts, so those numbers must be captured before hole merging changes the topology.

\subsection{Parser Module}

\code{src/parser.cpp} is responsible for converting assignment-format coordinate input into the half-edge gallery representation. Its strongest contribution is the orientation normalization step. Without that invariant, the later sweep-line predicates would become unreliable.

\subsection{Polygon Utility Module}

\code{src/polygon_utils.cpp} is small, but it supplies the cross-cutting traversal routines that keep the rest of the code clean. In particular, the hole merger depends on \code{getTopmostVertices} and the guard stage depends on \code{getTotalVertexCount}.

\subsection{Hole Merger}

\code{src/hole_merger.cpp} is the most delicate topological module in the submission. In the current version it combines geometric ordering, angular indexing, component tracking, same-level helper locking, bridge-target disambiguation, and local pointer rewiring. If this file is wrong, the rest of the pipeline cannot recover because every later stage assumes the merged boundary walk is valid.

\subsection{Triangulation Engine}

\code{src/triangulation.cpp} is the densest algorithmic file. It contains:

\begin{itemize}
\item occurrence-level polygon construction,
\item vertex classification,
\item monotone decomposition,
\item face extraction by synthetic outerplanar embedding,
\item monotone-face triangulation,
\item final triangle filtering and printing support.
\end{itemize}

The face-extraction logic is the most distinctive part of the solver. It is also the part that explains why the report spends so much time on occurrence indices and circular embeddings.

\subsection{Guard Module}

\code{src/guards.cpp} converts triangulation output into a graph problem and then back into geometric guard coordinates. The code is concise, but it correctly captures the theoretical idea: 3-color the triangulation graph and choose the smallest color class.

\subsection{Build and Run Layer}

\code{Makefile} compiles all source files in \code{src/} into \code{build/art_gallery}. \code{test.sh} is intentionally thin: it builds the solver and runs it on the supplied input. That separation keeps the build interface simple while the algorithmic complexity stays inside the C++ modules.

\section{Conclusion}

The current cleaned submission is a genuine end-to-end solution to the art gallery assignment with holes. Its overall structure is strong for three reasons:

\begin{itemize}
\item it keeps the topology manipulation localized in one bridge-insertion phase,
\item it uses occurrence-level indexing to handle repeated bridge endpoints cleanly,
\item it turns the final guard computation into a well-structured graph-coloring problem.
\end{itemize}

The most technically interesting parts of the current code are the component-aware helper locking inside the hole merger, the local bridge splice, the face extraction based on a synthetic circular embedding, and the way the guard stage chooses an actual guard set that can be smaller than the theoretical bound. The worked examples show that the same pipeline handles convex, concave, one-hole, and multi-hole galleries correctly.
